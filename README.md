# Melody Incipit Grid Tool

**Catch the melody before it disappears.**

The Melody Incipit Grid Tool is a single-page web app for capturing the *fingerprint* of a melodic idea — its incipit — as a stack of aligned musical lanes: **chords, beat counts, scale degrees, and lyrics/syllables**, plus searchable metadata and an audio reference.

It is *structured like a grid, used like a card, stored like data*: the entry surface is a stack of musical bar cards (never a spreadsheet), the canonical data is a per-subdivision cell grid (never preformatted strings), and the human-readable aligned output is generated by the system — you never fight spacing by hand.

```
Chords:  | C               | Bb              | F               | C               |
Counts:  | 1 &   2 &     3 & 4 &  | 1       & 2 & 3    & 4 & | ...
Degrees: | 3     3 2     1     5  | ♭7        6     5         | ...
Lyrics:  | I     was walk-ing home | through   the   rain      | ...
```

## The problem it solves

Melodic ideas evaporate. Voice memos preserve the *feel* but not the architecture; sheet music preserves the architecture but is too slow for capture; a spreadsheet aligns everything but feels sterile and makes *you* manage the grid. This tool captures both halves of an idea — pair it with an audio reference for the feel, and encode the structure in scale degrees fast enough to use in the moment, durable enough to rebuild the melody later and feed a future searchable song-idea corpus.

## Core workflow

```
Sing the idea → capture audio (voice memo / Logic)
            → open this tool
            → fill metadata (title, key, mode, meter, feel, tags…)
            → enter the melody into bar cards
              (chord per bar; degree + syllable per subdivision)
            → watch the aligned preview build itself
            → copy or download Markdown / JSON / plain text
            → file it in your song idea bank
```

## Running it locally

Serve the folder with any static file server. Using Python (preinstalled on macOS and most Linux systems):

```bash
cd /path/to/melody-incipit-tool
python3 -m http.server 8080
open http://localhost:8080/
```

If port 8080 is already in use, use 8081:

```bash
cd /path/to/melody-incipit-tool
python3 -m http.server 8081
open http://localhost:8081/
```

(On Linux, replace `open` with `xdg-open`; on Windows, `start`.) Stop the server with `Ctrl+C`.

To review the app: open `http://localhost:8080/` — that serves `index.html`, which is the whole application. To run the test suite, open `http://localhost:8080/tests.html`.

> Opening `index.html` directly from the file system also works for this tool, but the served method above is the supported review path and matches how GitHub Pages will serve it.

## Using the tool

### Melody metadata

The collapsible panel at the top holds everything that makes an incipit findable later:

| Field | Notes |
|---|---|
| Title | Human name for the idea |
| Song ID | Connects it to a larger song project |
| Key / Tonal center | Anchors the degree numbers (suggestions offered, free text allowed) |
| Mode / Scale | major, natural minor, blues, mixolydian, … (suggestions + free text) |
| Meter | 4/4, 3/4, 2/4, 6/8 fully gridded; any other value accepted (see below) |
| Tempo, Feel | straight, swing, gospel sway, 6/8 ballad, … |
| Hook function | opening line, chorus hook, verse motif, call-and-response, … |
| Status | raw, promising, hook, verse, chorus, bridge, motif, developed, archived |
| Audio reference | Voice memo name, Logic project, or a link |
| Tags | Comma-separated; rendered as chips and exported as a list |
| Contour, Notes | Free text about the phrase shape and anything else |

`date_created` / `date_modified` are handled automatically.

### Bars, lanes, and the grid

- **Phrase → Bars → Lanes.** An incipit holds one or more labeled phrases; each phrase is a stack of bar cards; each card has a chord at the top and three lanes beneath: system-generated **Counts**, editable **Degrees**, editable **Lyrics**.
- **Grid resolution** (Quarter / Eighth / Sixteenth) sets the subdivision of every bar. Counts regenerate automatically: `1 2 3 4`, `1 & 2 & 3 & 4 &`, or `1 e & a 2 e & a …`.
- **Switching resolution is safe.** Coarse → fine is lossless (your entries land on the matching slots). Fine → coarse warns you first and lists exactly what would be dropped; cancel keeps everything.
- **Meters.** 4/4, 3/4, and 2/4 grid as you'd expect. 6/8 is treated as a compound meter: at the eighth grid you get six pulses (`1 2 3 4 5 6`), at sixteenth each pulse splits (`1 & 2 & …`), and at the quarter (beat-level) grid you get the two dotted-quarter beats. Any other meter (say `7/8`) is accepted and stored, but bars use a 4/4-style grid and a notice tells you so.
- **Degrees.** Type `1`–`7`, accidentals as plain ASCII — `b3`, `#4`, `b7` — which render as ♭3, ♯4, ♭7 when you leave the cell (storage stays ASCII so the JSON is clean to type and grep). Use `-` for a held note; leave a cell empty for space/rest.
- **Lyrics / syllables.** Plain text per cell; hyphenate syllables as you like (`walk-`, `ing`). A long syllable widens its whole column in the output — alignment never breaks.
- **Bar management.** Per card: **Duplicate** (deep copy), **Clear** (empty the cells, keep the bar), **Delete** (with a 6-second Undo toast), and **◀ ▶** to reorder. **+ Add bar** sits under each phrase. Deleting the only bar gives you a fresh empty one — the editor never reaches zero bars.
- **Phrases.** **+ Add phrase** creates a new labeled group of bars; click a phrase label to rename it; phrases beyond the first can be deleted (with confirmation when they have content).

### Capture mode and Read mode

The header toggle switches between:

- **Capture** — the working mode: metadata, bar cards, live preview, exports.
- **Read** — the clean output: title, tonal summary, the full-width aligned grid, contour, notes, and tags, styled like a cue card. This is also exactly what **Print read view** prints.

### Live preview

The right-hand panel (below the editor on small screens) regenerates as you type, with three tabs:

- **Text** — the aligned plain-text incipit (character-identical to the plain-text export)
- **Markdown** — the full Markdown document with YAML frontmatter
- **JSON** — the canonical structured data

### Keyboard controls

The fast-capture path is designed so a whole melody can be entered without touching the mouse (the **Keys** button in the header shows this list in-app):

| Key | Action |
|---|---|
| `Tab` / `Shift+Tab` | Next / previous cell **along the lane**, flowing across bars (all degree cells first, then all lyric cells) |
| `Enter` | Jump to the next bar's first degree cell |
| `↑` / `↓` | Switch between the degree and lyric lanes in the same column |
| `←` / `→` | Move along the lane when the cursor is at the cell edge |
| `Ctrl/Cmd + D` | Duplicate the current bar |
| `Ctrl/Cmd + Enter` | Add a bar to the current phrase |

All buttons and controls are reachable with the keyboard, focus is always visible, and the preview announces politely to screen readers.

## The canonical data structure

The melody is **never** stored as display strings. The source of truth is per-subdivision cell arrays — the aligned text is generated from them. Full schema: [`schema/melody-incipit.schema.json`](schema/melody-incipit.schema.json).

```json
{
  "schema_version": "1.0",
  "type": "melody-incipit",
  "id": "2026-06-11-a1b2",
  "title": "Walking Home Hook",
  "song_id": "walking-home",
  "date_created": "2026-06-11",
  "date_modified": "2026-06-11",
  "audio_ref": "voice-memo-2026-04-29-214pm.m4a",
  "key": "C",
  "tonal_center": "C",
  "mode": "major",
  "meter": "4/4",
  "tempo": "",
  "feel": "gospel sway",
  "hook_function": "chorus hook",
  "default_grid_unit": "eighth",
  "contour": "Starts on 3, steps down to 1, drops to b7 color, resolves home.",
  "notes": "Keep the b7 against the Bb chord.",
  "tags": ["gospel", "melody", "hook", "flat-seven"],
  "status": "promising",
  "phrases": [
    {
      "label": "Phrase 1",
      "bars": [
        {
          "meter": "4/4",
          "grid_unit": "eighth",
          "chords": [{ "symbol": "C", "position": 0 }],
          "degrees": ["3", "", "3", "2", "1", "", "5", ""],
          "lyrics":  ["I", "", "was", "walk-", "ing", "", "home", ""]
        }
      ]
    }
  ]
}
```

Design notes baked into the schema:

- `degrees` and `lyrics` are parallel arrays whose length equals the slot count implied by `meter` × `grid_unit`.
- The **counts lane is derived, not stored** — it's a pure function of meter and grid unit, so it can never drift out of sync.
- `chords` is an array of `{symbol, position}` even though v1 always has one chord at position 0 — chord-per-beat becomes a non-breaking change later.
- `grid_unit` is stored **per bar** (inheriting the document default) so per-bar resolution and mixed meters are possible later without a migration.
- Accidentals are stored ASCII (`b3`, `#4`) and rendered as ♭/♯ by the display layer and text exports.

## Export formats

All exports are generated from the canonical data by pure functions; the aligned grid block is produced by **one renderer** shared by the preview, the plain-text export, and the Markdown export, so all three are guaranteed identical.

| Format | Buttons | Contents | Filename |
|---|---|---|---|
| **Markdown** | Copy Markdown / Download .md | YAML frontmatter (all metadata, `type: melody-incipit`) + readable body: audio reference, tonal setup, the aligned grid in a fenced code block, contour, notes. Parses cleanly as Obsidian frontmatter. | `YYYY-MM-DD-title-slug.md` |
| **JSON** | Copy JSON / Download .json | The canonical structured data — lossless, re-importable | `YYYY-MM-DD-title-slug.json` |
| **Plain text** | Copy plain text / Download .txt | One-line metadata header + the aligned grid + contour/notes/tags footer | `YYYY-MM-DD-title-slug.txt` |

**Print read view** opens the browser print dialog on the Read-mode sheet.

## Import

**Import JSON** (header) loads a previously downloaded `.json` export. Files are validated before loading — wrong type, missing phrases, mismatched lane lengths, or invalid grid units are rejected with a plain-language message and your current work is untouched. A successful import offers a one-step **Undo** back to what you had.

## Local persistence

- Every change autosaves (debounced ~0.4 s) to `localStorage` under `melody-incipit:current`; a quiet "Saved ✓" appears in the header. Reopening the page restores your work exactly.
- **New** asks first, then keeps your previous incipit under `melody-incipit:backup` *and* offers an immediate Undo toast.

### Known browser-storage limitations

`localStorage` is per-browser, per-device, and per-origin — and browsers may clear it (private windows, "clear site data," storage pressure, some content blockers). Treat it as a working buffer, not an archive: **Download JSON is the durable copy.** The single autosave slot holds one current incipit (plus one backup); a multi-incipit library is deliberately a future version — your filesystem/corpus is the library for now, via JSON/Markdown downloads and re-import.

## Creating a Git repository and deploying to GitHub Pages

These steps assume nothing exists yet — no local repo, no GitHub repo.

### 1. Initialize Git and make the first commit

```bash
cd /path/to/melody-incipit-tool
git init
git add .
git commit -m "Melody Incipit Grid Tool v0.1.0 — first complete functional build"
git branch -M main
```

### 2. Create and connect a GitHub repository

Create an empty repository on GitHub (no README, no .gitignore — the project already has its files): go to <https://github.com/new>, name it `melody-incipit-tool`, leave everything unchecked, and create it. Then connect and push:

```bash
git remote add origin https://github.com/YOUR-USERNAME/melody-incipit-tool.git
git push -u origin main
```

(If you use the GitHub CLI instead: `gh repo create melody-incipit-tool --public --source=. --push`.)

### 3. Enable GitHub Pages

1. On GitHub, open the repository → **Settings** → **Pages**.
2. Under **Build and deployment**, set **Source** to *Deploy from a branch*.
3. Choose branch **main**, folder **/ (root)**, and **Save**.
4. After a minute or two the tool is live at:

```
https://YOUR-USERNAME.github.io/melody-incipit-tool/
```

No build configuration is needed — the app is already static, dependency-free, and uses relative paths, so it works at any base URL.

### 4. Updating later

```bash
git add .
git commit -m "Describe the change"
git push
```

Pages redeploys automatically on push.

## Linking from the Chaos Portal

This tool is deliberately portal-independent: it has no hardcoded portal URLs, runs anywhere, and keeps a clean boundary with the wider ecosystem. The intended connection follows the same pattern the portal already uses for the **Piano Exercise Matrix** — an independently deployed app that the portal links to from a module card:

1. Deploy this tool to its own GitHub Pages URL (steps above).
2. In the Chaos Portal, add a module card / launcher entry pointing at that URL, with a title and one-line description in the portal's existing card style, e.g.:
   - **Melody Incipit Grid Tool** — *Capture melody fingerprints: chords, degrees, counts, and lyrics on an aligned grid.*
3. Nothing in this repository needs to change for that link to work; the portal entry is purely a pointer. If the portal later wants to *embed* the tool, an `<iframe>` of the deployed URL also works, since the app is self-contained on one page.

Because the data layer is a published JSON Schema with stable, cell-level structure, future ecosystem pieces (All Song App, Creativity Engine corpus) can consume the tool's exports directly — frontmatter'd Markdown for Obsidian-style corpora, JSON for anything programmatic.

## Limitations and future integration considerations

- One incipit at a time (single autosave slot + one backup). A saved-entries library with search/filter is the planned Version 2.
- CSV export, degree button palette, chord suggestions, quick-fill rhythms, per-bar grid resolution, and audio recording/playback are intentionally deferred (see `CHANGELOG.md`).
- 6/8 uses pulse-numbered counting (`1 2 3 4 5 6`) rather than `1 & a 2 & a`; meters outside 4/4 · 3/4 · 2/4 · 6/8 fall back to a 4/4-style grid with a notice.
- The ♭/♯ glyphs in text exports occupy one monospace column in most editors/fonts; a rare font may render them at odd widths. The underlying JSON is always ASCII.
- Very long chord names or syllables widen their bar in the text output (by design — alignment is never sacrificed); extremely long content simply makes a wide, horizontally scrollable line.

## Tests

Open `tests.html` (served the same way as the app) to run the assertion suite covering grid math for every supported meter × resolution, remapping (lossless up, warned-loss down, round trips), degree rendering, the alignment renderer (equal-width lanes, long-cell behavior, multi-phrase blocks), all three exporters, filename conventions, validation, and export → import round-trip integrity.
